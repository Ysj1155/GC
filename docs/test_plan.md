# SSD GC Validation Test Plan

This document describes the validation direction for the SSD GC Validation Lab. The goal is to make the repository useful as a storage validation / test engineering portfolio project, not only as a GC policy experiment.

## Scope

The project validates a simplified page-mapped SSD model with:

- Out-of-place host writes
- LPN to PPN mapping and reverse mapping
- VALID / INVALID / FREE page states
- TRIM invalidation
- Garbage collection and valid-page migration
- Wear tracking through block erase counts
- GC policy comparison through shared metrics

The model is intentionally simplified and does not claim real-device timing accuracy.

## Test Categories

### Functional Invariants

These tests check that the simulator state stays internally consistent.

- Page accounting: `valid + invalid + free == total_pages`
- Block accounting: each block's counters match its page states
- Mapping consistency: every LPN mapping has a matching reverse-map entry
- Reverse-map consistency: every reverse-map entry points back to the same LPN
- Mapped physical pages are always VALID

### Write / Overwrite Behavior

These tests check page-mapped FTL behavior.

- A new host write creates one VALID page
- Rewriting the same LPN invalidates the old physical page
- Host writes increment both host and device write counters
- Device writes are never lower than host writes

### TRIM Behavior

These tests check logical delete handling.

- TRIM removes the LPN mapping
- TRIM invalidates the previously mapped physical page
- TRIM does not increment host or device writes
- TRIM counters distinguish hits, misses, retrims, and invalidated pages
- Per-TRIM event logs preserve mapping invalidation evidence
- Repeated TRIM on an unmapped LPN is safe

### Garbage Collection Behavior

These tests check GC correctness and metric behavior.

- GC migrates valid pages before erasing a victim block
- GC preserves mapping / reverse-map consistency
- GC increments erase count and GC count
- All-invalid victim blocks are still counted as GC events
- WAF remains greater than or equal to 1 after host writes

### Reproducibility

These tests check deterministic behavior under fixed seeds.

- Workload generation is deterministic for the same seed
- Simulation metrics are reproducible for the same config, workload, and policy
- Different seeds may produce different behavior, but all invariant checks must still pass

### Stress / Endurance Matrix

These tests are planned as portfolio validation scenarios.

- High update ratio random write stress: `random_update_stress`
- Hot/cold skew stress: included in update-heavy presets
- TRIM burst stress: `trim_burst`
- Delete-after-bulk-load TRIM lifecycle stress: `delete_after_bulk_load`
- Low over-provisioning pressure: `low_op_pressure`
- Multi-policy comparison under identical workload seeds: `validation_matrix.py`
- Long-run endurance wear distribution: `endurance_short`

The quick matrix is intended for local smoke validation. The full matrix is intended for final portfolio artifacts:

```bash
python tools/validation_matrix.py --profile quick --policies greedy,cota --seeds 41 --dry_run
python tools/validation_matrix.py --profile full --policies greedy,cb,bsgc,cota --out_dir results/final_clean
```

## Pass / Fail Signals

The core pass/fail signals are:

- No invariant violations
- No unmapped or stale reverse-map entries
- `WAF >= 1` whenever host writes are nonzero
- Page totals match physical capacity
- GC count and erase count are consistent with GC activity
- Repeated runs with the same seed produce the same summary metrics

## Reproducibility Artifacts

Every portfolio-grade validation run should keep a manifest JSON generated by `run_sim.py --manifest_json`.

The manifest should capture:

- command-line parameters
- output artifact paths
- Python executable and version
- git branch, commit, and dirty working-tree status
- final metrics row
- creation timestamp

For final portfolio results, prefer one clean output directory per scenario, such as `results/final_clean/greedy_41/manifest.json`, instead of appending unrelated experiments into a shared folder.

## Report Generation

After running the validation matrix, generate a reviewable report:

```bash
python tools/validation_report.py --base_dir results/final_clean
```

Expected outputs:

- `validation_report.md`
- TRIM lifecycle evidence: `docs/trim_gc_validation_evidence.md`
- `validation_summary.csv`
- `validation_runs.csv`

## Portfolio Notes

For interview review, the most important artifacts are:

- `tests/` for executable validation intent
- `run_sim.py --qc strict` for runtime sanity checks
- Clean result manifests for reproducible experiment sets
- A short validation report summarizing workload, policy matrix, pass/fail status, and key metrics
## TRIM-to-GC lag analysis

- TRIM hit rows that invalidate a physical page are eligible for lag tracking.
- The analyzer matches each eligible TRIM row to the next GC event whose victim block is the same physical block.
- Reclaimed rows record `lag_steps`; pending rows remain visible with an empty lag.
- Summary metrics include eligible, reclaimed, pending, reclaim rate, average lag, p95 lag, and max lag.
## TRIM window analysis

- Trace rows include free blocks, valid pages, invalid pages, host writes, device writes, GC count, and TRIM count.
- TRIM events produced by Simulator include workload `op_step` for trace alignment.
- Nearby TRIM events are grouped into windows using `trim_window_merge_gap`.
- Each window compares before/after snapshots and records invalid page, free block, GC count, and WAF deltas.
- Summary metrics include window count, average trim ops, average deltas, and windows with GC activity.
## TRIM locality sensitivity

- Matrix scenarios include hot, cold, and mixed TRIM locality variants with matched workload parameters.
- Report rows preserve `trim_locality` for per-run inspection.
- Locality summary groups runs by policy and TRIM target locality.
- Sensitivity outputs compare WAF, GC count, wear spread, lag, reclaim rate, and window movement deltas.
## Focused matrix filters

- `validation_matrix.py --scenarios` can restrict a portfolio run to selected scenarios.
- Scenario filters preserve the requested order and fail on unknown names.
- TRIM evidence runs should use focused filters when collecting multi-policy, multi-seed artifacts.
## TRIM evidence matrix

- Focused evidence run covers TRIM burst, delete-after-bulk-load, and hot/cold/mixed locality scenarios.
- The matrix crosses four policies and three seeds for 60 strict-QC runs.
- Evidence outputs include validation report, validation summary, locality summary, insight packet, and LLM narrative packet.
- The run is documented in `docs/trim_evidence_matrix.md`.
